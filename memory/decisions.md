## 2026-07-20 · Researched real agent-memory token-efficiency practice and added a Gist column, deliberately declining vector search
**Decision:** Researched how Mem0, Letta (MemGPT), and Zep hit their published token-efficiency numbers (Mem0: ~1.8K tokens/query vs ~26K full-context on LOCOMO): compress at write time, let a cheap index resolve most lookups via embeddings and a vector store. Adopted the principle, not the mechanism: added an optional `Gist` column to `memory/INDEX.md`, one line per entry, so most lookups resolve from the index without opening `decisions.md`. Backfilled all 15 existing entries. Declined embeddings/vector search: it's a dependency and a service (violates zero-dependency, no-network), and semantic similarity is approximate by construction, disqualifying for a system whose premise is exact proof, not fuzzy recall. Rewrote index parsing (`parse_index_table`) to be column-name driven instead of hardcoded `cells[2]`/`cells[3]`, so Gist didn't break Title/token parsing. Found and fixed two real regressions the rewrite caused: a too-strict version dropped a phantom row appended past a table's end (fixed with a positional fallback); and it exposed a genuine pre-existing gap in `check_handoffs`, which used to skip entirely when zero real handoff files existed, before ever checking whether INDEX.md's own handoffs table claimed a phantom one anyway (fixed to check regardless). Full detail on both fixes lives in the code comments at `parse_index_table` and `check_handoffs` in `molt-verify.py`. 4 new adversarial cases; all 36 pass at n=300 and n=3000. Non-adoption documented in `SCALE-SPEC.md`.
**Reasoning:** Requested directly: research what others do and incorporate what's genuinely useful. The useful part was the principle (compress once, verify honesty, let the index carry most lookups), not the technology; Molt already had progressive disclosure, just without a rich-enough index entry to resolve most lookups without a full fetch. The two masked gaps surfaced by tightening the parser are the same lesson this project keeps relearning: an accidental catch isn't a real one.
**Reversible:** Yes. Gist is opt-in, same pattern as the mirror check; `parse_index_table` is a strict superset of the old parser's coverage; the `check_handoffs` fix only adds detection where it previously silently skipped.
**Review:** Gist quality is a human judgment; `molt-verify.py` can only check "is it present," not "is it accurate." Revisit with a human spot-check if gists start drifting from what entries actually say. This entry itself was rewritten once, before hashing, to fit the 600-token budget the previous entry just introduced; that's the discipline working as intended, not an exception to it.

## 2026-07-20 · Added hash-chained tamper-evidence for memory/decisions.md
**Decision:** Every entry can now carry an optional `**Hash:**` field: sha256(that entry's own required-field text + the previous, older entry's hash), oldest to newest, genesis hash all zeros. Added `check_hash_chain` to `molt-verify.py`: skips if no entries have a Hash field (opt-in, like the mirror check); fails if some entries have one and others don't (an incomplete chain gives false confidence); verifies the full chain and fails at the first break if any historical entry's hash doesn't match its recomputed value. Wrote `molt-chain-append.py`, a small standalone tool that backfills or extends the chain by pure deterministic hashing, never judging or summarizing content, consistent with the "human writes every decision" principle. Backfilled all 9 real entries in this project's own log. Verified the mechanism catches: tampering with the oldest entry (breaks the chain), and a chain missing one entry's hash (fails as incomplete). Added both as permanent adversarial cases, plus a hash_chain_control case, using a separate "chained" synthetic base so the other 19 cases stay unaffected. All 22 cases pass at n=300 and n=3000 (1.76s).
**Reasoning:** Verified this against a real, dated standard rather than inventing a scheme: the IETF's March 2026 "Agent Audit Trail" draft defines SHA-256 hash-chained records for autonomous AI systems, explicitly to satisfy EU AI Act Article 12's automatic event-logging requirement for high-risk systems, effective August 2026, this month. Before this, Molt's only tamper detection required maintaining an external mirror; an unmirrored log could have an old entry edited in place with nothing catching it. This closes that gap without requiring a mirror at all, the single biggest remaining trust hole this project had.
**Reversible:** Yes. The Hash fields are additive lines within existing entries; deleting `molt-chain-append.py` and the Hash fields returns the log to its prior, still-valid shape. `check_hash_chain` no-ops cleanly if no entries have a hash.

Revisit if that turns out to happen in practice; it hasn't yet, and being strict by default is the safer failure mode for a tamper-evidence feature.
